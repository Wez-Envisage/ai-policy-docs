Section 4

Cross-Verifying with Gemini & ChatGPT

Every model has blind spots, and our deliverables carry our name to UN agencies, funds, and corporate clients. A second model is a cheap independent check — when used on the right things, the right way.

The Principle

Cross-verification means taking a specific claim or output produced with one model and independently checking it with another (Gemini or ChatGPT). It is not re-doing the whole task twice. Two rules govern every verification:

1

Verify the claim, not the conversation

Paste only the sentence, figure, or code block being checked — never the source document, the chat history, or surrounding analysis. This protects confidentiality and costs a fraction of the tokens.

2

Tier 2 maximum, sanitized

Verification tools never receive Tier 3 content. Anonymize first (see Data Security). If a claim can't be checked without revealing restricted detail, verify it against primary sources by hand instead.

Agreement is not truth. Two models can share the same wrong answer, especially on recent events or thin-data regions like those we cover. Model agreement raises confidence; only a primary source settles a fact destined for a client deliverable.

When to Verify — and With What

SituationVerify?How
Statistic or factual claim going into a client deliverableAlwaysAsk Gemini or ChatGPT to confirm/refute with sources; then check the primary source yourself.
Translation of source material (Burmese, Thai, Shan, etc.) affecting a findingAlwaysSecond model translates the same sanitized passage blind; compare meanings. Disagreement → human translator.
Legal, regulatory, or sanctions-related statementsAlwaysCross-check, then confirm against the official text. Never deliver on model output alone.
SQL or Python destined for the Sapphire pipelineUsuallyPaste the code (schema names only, no data) into a second model: "Find bugs and edge cases." Then test on dev data.
Summary of a public documentSpot-checkVerify the 2–3 claims you'd be most embarrassed to get wrong.
Internal brainstorm, draft wording, formattingNoEditing judgment, not facts. Verification here is token waste.

Worked Examples

Example 1 — Statistic in a deliverable

Scenario

Claude drafts: "The top ten industrialized nations spend a combined ~$200 billion annually on development and foreign aid." This is headed for a client report.

To Gemini: "What is the most recent combined ODA spending of the top ten donor nations? Cite the source." If Gemini's figure differs materially, pull the OECD data directly and use the primary number with a citation. Cost: ~100 tokens. Value: our credibility.

Example 2 — Translation check

Scenario

A harvested Burmese-language article appears to say a company "acquired" a concession; the finding hinges on whether it means acquired or applied for.

To ChatGPT (sanitized passage, names replaced): "Translate this passage precisely. Does the verb indicate completed acquisition or a pending application?" If the models disagree, the passage goes to a human translator on the team before the finding is written.

Example 3 — SQL for the pipeline

Scenario

Claude wrote a Snowflake query joining three tables to flag entities appearing across datasets.

To Gemini: paste the query plus a one-line schema description: "Review for logic errors, join fan-out, and NULL handling." Models are excellent at catching each other's off-by-one logic. Then run on dev data before production.

Example 4 — When NOT to verify

Scenario

Claude restructured your interview notes into themes. There is no external fact to check — review it yourself. Sending it to a second model adds token cost and a second exposure point for zero gain.

Verification Etiquette (Token-Efficient)

  • Blind the second model. Don't say "Claude said X — is that right?" Ask the question fresh. Models anchor on suggested answers.
  • Batch your checks. Collect the deliverable's 5 key claims and verify them in one message.
  • Use free tiers for verification. Verification prompts are short and need no context — ideal for free-tier Gemini/ChatGPT, preserving paid capacity.
  • Record what you verified. One line in the project log: claim, tools used, primary source. This is our audit trail — the same discipline SAM sells to fund managers.

หมวดที่ 4

การตรวจสอบไขว้ด้วย Gemini และ ChatGPT

ทุกโมเดลมีจุดบอด และงานส่งมอบของเราถือชื่อเราไปสู่หน่วยงาน UN กองทุน และลูกค้าองค์กร โมเดลที่สองคือการตรวจสอบอิสระราคาถูก — เมื่อใช้กับสิ่งที่ถูกต้อง ด้วยวิธีที่ถูกต้อง

หลักการ

การตรวจสอบไขว้คือการนำข้อความหรือผลลัพธ์เฉพาะที่สร้างด้วยโมเดลหนึ่ง ไปตรวจสอบอย่างอิสระกับอีกโมเดล (Gemini หรือ ChatGPT) ไม่ใช่การทำงานทั้งหมดซ้ำสองรอบ กฎสองข้อกำกับทุกการตรวจสอบ:

1

ตรวจข้อความ ไม่ใช่บทสนทนา

วางเฉพาะประโยค ตัวเลข หรือโค้ดที่ต้องตรวจ — ห้ามวางเอกสารต้นทาง ประวัติแชต หรือบทวิเคราะห์รอบข้าง ปกป้องความลับและประหยัดโทเคนมหาศาล

2

สูงสุด Tier 2 และคัดกรองแล้ว

เครื่องมือตรวจสอบห้ามรับเนื้อหา Tier 3 ปกปิดตัวตนก่อน (ดู ความปลอดภัยข้อมูล) ถ้าตรวจไม่ได้โดยไม่เปิดเผยรายละเอียดหวงห้าม ให้ตรวจกับแหล่งข้อมูลปฐมภูมิด้วยตนเองแทน

การเห็นพ้องไม่ใช่ความจริง สองโมเดลอาจให้คำตอบผิดเหมือนกัน โดยเฉพาะเหตุการณ์ล่าสุดหรือภูมิภาคที่ข้อมูลเบาบางอย่างพื้นที่ที่เราดูแล การเห็นพ้องของโมเดลเพิ่มความเชื่อมั่น แต่เฉพาะแหล่งข้อมูลปฐมภูมิเท่านั้นที่ชี้ขาดข้อเท็จจริงที่จะเข้างานส่งมอบลูกค้า

เมื่อไรต้องตรวจ — และด้วยอะไร

สถานการณ์ตรวจไหมวิธี
สถิติหรือข้อเท็จจริงที่จะเข้างานส่งมอบลูกค้าเสมอให้ Gemini หรือ ChatGPT ยืนยัน/หักล้างพร้อมแหล่งอ้างอิง แล้วตรวจแหล่งปฐมภูมิเอง
คำแปลของเอกสารต้นทาง (พม่า ไทย ไทใหญ่ ฯลฯ) ที่กระทบข้อค้นพบเสมอให้โมเดลที่สองแปลข้อความเดียวกัน (คัดกรองแล้ว) แบบไม่เห็นคำแปลแรก แล้วเทียบความหมาย ถ้าขัดแย้ง → นักแปลมนุษย์
ข้อความด้านกฎหมาย กฎระเบียบ หรือมาตรการคว่ำบาตรเสมอตรวจไขว้ แล้วยืนยันกับเอกสารทางการ ห้ามส่งมอบโดยอาศัยผลลัพธ์โมเดลอย่างเดียว
SQL หรือ Python สำหรับไปป์ไลน์ Sapphireมักจะตรวจวางโค้ด (เฉพาะชื่อ schema ไม่มีข้อมูล) ในโมเดลที่สอง: "หาบั๊กและกรณีขอบ" แล้วทดสอบกับข้อมูล dev
บทสรุปของเอกสารสาธารณะสุ่มตรวจตรวจ 2–3 ข้อความที่จะอายที่สุดถ้าผิด
ระดมความคิดภายใน ร่างถ้อยคำ จัดรูปแบบไม่เป็นวิจารณญาณการแก้ไข ไม่ใช่ข้อเท็จจริง การตรวจตรงนี้คือเปลืองโทเคน

ตัวอย่างประกอบ

ตัวอย่าง 1 — สถิติในงานส่งมอบ

สถานการณ์

Claude ร่างว่า: "สิบประเทศอุตสาหกรรมชั้นนำใช้จ่ายรวมราว 2 แสนล้านดอลลาร์ต่อปีกับการพัฒนาและความช่วยเหลือต่างประเทศ" ข้อความนี้กำลังเข้ารายงานลูกค้า

ถาม Gemini: "ยอด ODA รวมล่าสุดของสิบประเทศผู้บริจาคชั้นนำคือเท่าไร อ้างอิงแหล่งที่มา" ถ้าตัวเลขต่างกันมาก ให้ดึงข้อมูล OECD โดยตรงและใช้ตัวเลขปฐมภูมิพร้อมการอ้างอิง ต้นทุน: ~100 โทเคน มูลค่า: ความน่าเชื่อถือของเรา

ตัวอย่าง 2 — ตรวจคำแปล

สถานการณ์

บทความภาษาพม่าที่เก็บมาดูเหมือนบอกว่าบริษัท "ได้รับ" สัมปทาน ข้อค้นพบขึ้นอยู่กับว่าหมายถึงได้รับแล้วหรือแค่ยื่นขอ

ถาม ChatGPT (ข้อความคัดกรองแล้ว เปลี่ยนชื่อ): "แปลข้อความนี้อย่างแม่นยำ คำกริยาบ่งบอกการได้รับเสร็จสิ้นหรือคำขอที่รออนุมัติ" ถ้าโมเดลขัดแย้งกัน ส่งให้นักแปลมนุษย์ในทีมก่อนเขียนข้อค้นพบ

ตัวอย่าง 3 — SQL สำหรับไปป์ไลน์

สถานการณ์

Claude เขียนคิวรี Snowflake เชื่อมสามตารางเพื่อหาเอนทิตีที่ปรากฏข้ามชุดข้อมูล

ถาม Gemini: วางคิวรีพร้อมคำอธิบาย schema หนึ่งบรรทัด: "ตรวจหาข้อผิดพลาดเชิงตรรกะ join fan-out และการจัดการ NULL" โมเดลเก่งมากในการจับตรรกะพลาดของกันและกัน แล้วรันกับข้อมูล dev ก่อน production

ตัวอย่าง 4 — เมื่อไรไม่ต้องตรวจ

สถานการณ์

Claude จัดบันทึกสัมภาษณ์ของคุณเป็นธีม ไม่มีข้อเท็จจริงภายนอกให้ตรวจ — ตรวจเองได้เลย การส่งให้โมเดลที่สองเพิ่มต้นทุนโทเคนและจุดเสี่ยงข้อมูลโดยไม่ได้อะไร

มารยาทการตรวจสอบ (แบบประหยัดโทเคน)

  • ไม่บอกใบ้โมเดลที่สอง อย่าถามว่า "Claude บอกว่า X ถูกไหม" ให้ถามคำถามใหม่ตั้งแต่ต้น โมเดลมักคล้อยตามคำตอบที่ถูกชี้นำ
  • รวมการตรวจเป็นชุด รวบรวม 5 ข้อความสำคัญของงานส่งมอบ แล้วตรวจในข้อความเดียว
  • ใช้รุ่นฟรีสำหรับการตรวจสอบ พรอมต์ตรวจสอบสั้นและไม่ต้องใช้บริบท — เหมาะกับ Gemini/ChatGPT รุ่นฟรี เก็บความจุแบบเสียเงินไว้
  • บันทึกสิ่งที่ตรวจแล้ว หนึ่งบรรทัดในบันทึกโครงการ: ข้อความ เครื่องมือที่ใช้ แหล่งปฐมภูมิ นี่คือหลักฐานการตรวจสอบของเรา — วินัยเดียวกับที่ SAM ขายให้ผู้จัดการกองทุน

အပိုင်း ၄

Gemini နှင့် ChatGPT ဖြင့် ပြန်လှန်စစ်ဆေးခြင်း

မော်ဒယ်တိုင်းတွင် မမြင်နိုင်သောအားနည်းချက်များရှိပြီး ကျွန်ုပ်တို့၏ပေးအပ်စာတမ်းများသည် ကုလသမဂ္ဂအေဂျင်စီများ၊ ရန်ပုံငွေများနှင့် ကော်ပိုရိတ်ဖောက်သည်များထံ ကျွန်ုပ်တို့၏နာမည်ဖြင့် ရောက်သွားသည်။ ဒုတိယမော်ဒယ်တစ်ခုသည် စျေးသက်သာသော လွတ်လပ်သည့်စစ်ဆေးမှုဖြစ်သည် — မှန်ကန်သောအရာများအပေါ် မှန်ကန်သောနည်းဖြင့် သုံးလျှင်။

အခြေခံမူ

ပြန်လှန်စစ်ဆေးခြင်းဆိုသည်မှာ မော်ဒယ်တစ်ခုဖြင့်ထုတ်လုပ်ထားသော သီးခြားအချက် သို့မဟုတ် ရလဒ် ကို အခြားမော်ဒယ် (Gemini သို့မဟုတ် ChatGPT) ဖြင့် လွတ်လပ်စွာစစ်ဆေးခြင်းဖြစ်သည်။ အလုပ်တစ်ခုလုံးကို နှစ်ကြိမ်ပြန်လုပ်ခြင်း မဟုတ်ပါ။ စစ်ဆေးမှုတိုင်းကို စည်းမျဉ်းနှစ်ခုက အုပ်ချုပ်သည် —

1

စကားပြောမဟုတ်၊ အချက်ကိုသာ စစ်ပါ

စစ်ဆေးမည့် စာကြောင်း၊ ကိန်းဂဏန်း သို့မဟုတ် ကုဒ်အပိုင်းကိုသာ ကူးထည့်ပါ — မူရင်းစာတမ်း၊ စကားပြောမှတ်တမ်း သို့မဟုတ် ပတ်ဝန်းကျင်ခွဲခြမ်းစိတ်ဖြာချက်များ ဘယ်တော့မှ မထည့်ရ။ ဤနည်းက လျှို့ဝှက်မှုကိုကာကွယ်ပြီး တိုကင်အနည်းငယ်သာ ကုန်သည်။

2

အများဆုံး Tier 2၊ သန့်စင်ပြီး

စစ်ဆေးရေးကိရိယာများသည် Tier 3 အကြောင်းအရာကို ဘယ်တော့မှ မရရှိရ။ အရင်အမည်ဖျောက်ပါ (ဒေတာလုံခြုံရေး ကြည့်ပါ)။ ကန့်သတ်အသေးစိတ်မဖော်ဘဲ မစစ်နိုင်လျှင် မူရင်းရင်းမြစ်များဖြင့် ကိုယ်တိုင်စစ်ပါ။

သဘောတူညီမှုသည် အမှန်တရားမဟုတ်ပါ။ မော်ဒယ်နှစ်ခုသည် တူညီသောအဖြေမှားကို မျှဝေနိုင်သည် — အထူးသဖြင့် မကြာသေးမီဖြစ်ရပ်များ သို့မဟုတ် ကျွန်ုပ်တို့လုပ်ကိုင်သော ဒေတာနည်းပါးသည့်ဒေသများတွင်။ မော်ဒယ်သဘောတူညီမှုက ယုံကြည်မှုမြှင့်သည်; ဖောက်သည်ပေးအပ်စာတမ်းအတွက် အချက်တစ်ခုကိုမူ မူရင်းရင်းမြစ်ကသာ အဆုံးအဖြတ်ပေးသည်။

ဘယ်အချိန်စစ်မည် — ဘာနဲ့စစ်မည်

အခြေအနေစစ်မလားနည်းလမ်း
ဖောက်သည်ပေးအပ်စာတမ်းထဲဝင်မည့် စာရင်းအင်း သို့မဟုတ် အချက်အလက်အမြဲGemini သို့မဟုတ် ChatGPT ကို ရင်းမြစ်နှင့်တကွ အတည်ပြု/ငြင်းဆိုခိုင်းပါ; ထို့နောက် မူရင်းရင်းမြစ်ကို ကိုယ်တိုင်စစ်ပါ။
တွေ့ရှိချက်ကို သက်ရောက်သော မူရင်းဘာသာပြန် (မြန်မာ၊ ထိုင်း၊ ရှမ်း စသည်)အမြဲဒုတိယမော်ဒယ်ကို သန့်စင်ထားသောစာပိုဒ်တူကို သီးခြားဘာသာပြန်ခိုင်းပြီး အဓိပ္ပာယ်နှိုင်းယှဉ်ပါ။ ကွဲလွဲလျှင် → လူဘာသာပြန်။
ဥပဒေ၊ စည်းမျဉ်း သို့မဟုတ် ပိတ်ဆို့မှုဆိုင်ရာ ဖော်ပြချက်များအမြဲပြန်လှန်စစ်ပြီး တရားဝင်စာသားနှင့် အတည်ပြုပါ။ မော်ဒယ်ရလဒ်တစ်ခုတည်းဖြင့် ဘယ်တော့မှ မပေးအပ်ရ။
Sapphire pipeline အတွက် SQL သို့မဟုတ် Pythonများသောအားဖြင့်ကုဒ်ကို (schema နာမည်များသာ၊ ဒေတာမပါ) ဒုတိယမော်ဒယ်ထဲထည့်ပါ — "bug နှင့် edge case များရှာပါ"။ ထို့နောက် dev ဒေတာဖြင့် စမ်းပါ။
အများပြည်သူစာတမ်း အကျဉ်းချုပ်နမူနာစစ်မှားလျှင် အရှက်ရဆုံးဖြစ်မည့် အချက် ၂–၃ ခုကို စစ်ပါ။
အတွင်းပိုင်းစိတ်ကူးထုတ်ခြင်း၊ စကားလုံးမူကြမ်း၊ ပုံစံချခြင်းမစစ်ပါအချက်အလက်မဟုတ်၊ တည်းဖြတ်ဆင်ခြင်မှုသာ။ ဤနေရာတွင်စစ်ခြင်းသည် တိုကင်ဖြုန်းတီးမှုဖြစ်သည်။

လက်တွေ့နမူနာများ

နမူနာ ၁ — ပေးအပ်စာတမ်းထဲက စာရင်းအင်း

အခြေအနေ

Claude က ရေးသည် — "ထိပ်တန်းစက်မှုနိုင်ငံဆယ်ခုသည် ဖွံ့ဖြိုးရေးနှင့် နိုင်ငံခြားအကူအညီအတွက် နှစ်စဉ်စုစုပေါင်း ~ဒေါ်လာ ဘီလီယံ ၂၀၀ သုံးစွဲသည်"။ ၎င်းသည် ဖောက်သည်အစီရင်ခံစာထဲ ဝင်တော့မည်။

Gemini ကိုမေးပါ — "ထိပ်တန်းအလှူရှင်နိုင်ငံဆယ်ခု၏ နောက်ဆုံးစုစုပေါင်း ODA သုံးစွဲမှုက ဘယ်လောက်လဲ? ရင်းမြစ်ကိုးကားပါ။" Gemini ၏ကိန်းဂဏန်း သိသိသာသာကွဲလျှင် OECD ဒေတာကို တိုက်ရိုက်ဆွဲပြီး မူရင်းကိန်းဂဏန်းကို ကိုးကားချက်နှင့်တကွ သုံးပါ။ ကုန်ကျမှု — တိုကင် ~၁၀၀။ တန်ဖိုး — ကျွန်ုပ်တို့၏ ယုံကြည်စိတ်ချရမှု။

နမူနာ ၂ — ဘာသာပြန် စစ်ဆေးခြင်း

အခြေအနေ

စုဆောင်းထားသော မြန်မာဘာသာဆောင်းပါးတစ်ပုဒ်က ကုမ္ပဏီတစ်ခုသည် လုပ်ပိုင်ခွင့်ကို "ရရှိခဲ့သည်" ဟုဆိုပုံရသည်; ရရှိပြီးပြီလား လျှောက်ထားဆဲလားဆိုသည်အပေါ် တွေ့ရှိချက် မူတည်နေသည်။

ChatGPT ကို (သန့်စင်ထားသောစာပိုဒ်၊ နာမည်များအစားထိုးပြီး) — "ဤစာပိုဒ်ကို တိကျစွာဘာသာပြန်ပါ။ ကြိယာက ပြီးမြောက်သောရယူမှုလား ဆိုင်းငံ့ဆဲလျှောက်လွှာလား?" မော်ဒယ်များကွဲလွဲလျှင် တွေ့ရှိချက်မရေးခင် အဖွဲ့ထဲက လူဘာသာပြန်ထံ ပို့ပါ။

နမူနာ ၃ — Pipeline အတွက် SQL

အခြေအနေ

ဒေတာအစုများတစ်လျှောက် ပေါ်လာသော entity များကို သတ်မှတ်ရန် Claude က ဇယားသုံးခုပေါင်းသော Snowflake query ရေးခဲ့သည်။

Gemini ကို — query နှင့် schema ဖော်ပြချက်တစ်ကြောင်း ကူးထည့်ပါ — "Logic အမှားများ၊ join fan-out နှင့် NULL ကိုင်တွယ်မှုကို စစ်ပါ"။ မော်ဒယ်များသည် တစ်ခုနှင့်တစ်ခု၏ logic အမှားများကို ဖမ်းရာတွင် အလွန်တော်သည်။ ထို့နောက် production မတိုင်မီ dev ဒေတာဖြင့် run ပါ။

နမူနာ ၄ — ဘယ်အချိန် မစစ်သင့်လဲ

အခြေအနေ

Claude က သင်၏အင်တာဗျူးမှတ်စုများကို အကြောင်းအရာအလိုက် ပြန်ဖွဲ့စည်းပေးခဲ့သည်။ စစ်ရန်ပြင်ပအချက်အလက် မရှိပါ — ကိုယ်တိုင်ပြန်ကြည့်ပါ။ ဒုတိယမော်ဒယ်ထံပို့ခြင်းသည် ဘာအကျိုးမှမရဘဲ တိုကင်ကုန်ကျမှုနှင့် ဒုတိယပေါက်ကြားနိုင်သည့်နေရာကို ထပ်ထည့်ခြင်းသာဖြစ်သည်။

စစ်ဆေးရေး ကျင့်ဝတ် (တိုကင်ချွေတာသော)

  • ဒုတိယမော်ဒယ်ကို အမြင်မပေးပါနှင့်။ "Claude က X လို့ပြောတယ် — မှန်လား?" ဟု မမေးပါနှင့်။ မေးခွန်းကို အသစ်ကမေးပါ။ မော်ဒယ်များသည် အကြံပြုထားသောအဖြေများပေါ် မှီတတ်သည်။
  • စစ်ဆေးမှုများကို စုပေါင်းလုပ်ပါ။ ပေးအပ်စာတမ်း၏ အဓိကအချက် ၅ ခုကိုစုပြီး မက်ဆေ့ချ်တစ်ခုတည်းတွင် စစ်ပါ။
  • စစ်ဆေးရေးအတွက် အခမဲ့အဆင့်များသုံးပါ။ စစ်ဆေးရေး prompt များသည် တိုပြီး နောက်ခံအကြောင်းအရာမလိုပါ — အခမဲ့ Gemini/ChatGPT အတွက် အကောင်းဆုံးဖြစ်ပြီး အခပေးစွမ်းရည်ကို ထိန်းသိမ်းသည်။
  • စစ်ပြီးသမျှကို မှတ်တမ်းတင်ပါ။ ပရောဂျက်မှတ်တမ်းတွင် တစ်ကြောင်း — အချက်၊ သုံးသောကိရိယာများ၊ မူရင်းရင်းမြစ်။ ဤသည်မှာ ကျွန်ုပ်တို့၏ audit trail ဖြစ်သည် — SAM က ရန်ပုံငွေမန်နေဂျာများကို ရောင်းချသော စည်းကမ်းမျိုးပင်ဖြစ်သည်။
